## 前端修改任务规则

### 1. 修改共享组件时必须搜索所有使用方

当修改任何共享组件（如对话框、表单组件等）的 emit 签名、props 或行为时，必须：

1. 先搜索整个项目中所有引用该组件的文件（`views/`、`components/` 等）
2. 对每一个使用方都进行相应适配
3. 不得只修改部分使用方就标记任务完成

### 2. 修改类型/接口定义后必须同步修改所有调用处

当修改 `src/types/` 中的 TypeScript 接口或类型时，必须：

1. 搜索所有使用该类型的 API 函数、组件和视图
2. 同步更新所有调用处（传参、emit、表单字段等）
3. 不得只改类型定义而不改实际使用的地方

### 3. 对话框组件修改时需考虑初始状态的完整性

修改对话框组件时，需主动检查：

1. 打开对话框时的初始值是否完整、符合用户预期（例如：包含路径信息的文件名应显示完整路径）
2. 用户能否从初始状态直观了解当前操作对象的完整上下文

### 4. 任务完成前进行全量影响面检查

在标记任务完成前，必须自问：

- 是否有其他文件也依赖我修改的内容？
- 修改的组件是否在多处被使用？
- 类型修改是否影响到了所有调用链路？

只有确认所有受影响的文件都已处理完毕，才能标记任务完成。

### 5. 操作文件系统前先检查项目现有目录结构

在创建配置文件、规则文件、或任何"约定位置"的文件前，必须：

1. 先查看项目现有目录结构（如 `.roo/`、`.vscode/` 等），了解项目使用的规范和约定
2. 不得凭空猜测文件名或路径（例如误用 `.clinerules` 而非 `.roo/rules/rules.md`）
3. 若不确定正确位置，应查阅现有文件结构而非直接创建
